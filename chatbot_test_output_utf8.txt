### Limites  
- Les prototypes ne sont pas toujours prêts à être mis en production (manque d’optimisation, de tests, de monitoring).

## 📊 SQL : un « must‑have » ou un « nice‑to‑have » pour un Data Scientist ?  

### 1️⃣ Ce que fait SQL dans le quotidien d’un Data Scientist  

| **Fonction** | **Comment SQL intervient** | **Exemple concret** |
|--------------|----------------------------|----------------------|
| **Accès aux données** | Interroger les bases relationnelles (PostgreSQL, MySQL, Oracle, SQL‑Server, Snowflake, BigQuery, …) pour extraire les tables ou les sous‑ensembles pertinents. | `SELECT * FROM ventes WHERE date BETWEEN '2023‑01‑01' AND '2023‑12‑31';` |
| **Filtrage & agrégation** | Faire du *group‑by*, des *joins*, des *window functions* afin de préparer les variables (features) avant même de les charger dans Python/R. | `SELECT client_id, SUM(montant) AS total_achat FROM ventes GROUP BY client_id;` |
| **Nettoyage léger** | Supprimer les doublons, gérer les valeurs manquantes, transformer des formats (dates, chaînes). | `SELECT DISTINCT * FROM logs WHERE event IS NOT NULL;` |
| **Performance** | Déplacer le travail lourd (jointures, agrégations sur des millions de lignes) du notebook Python vers le moteur de la base, qui est optimisé pour cela. | Une agrégation de 100 M de lignes qui met 2 s en SQL vs 30 s en pandas. |
| **Documentation & traçabilité** | Les requêtes SQL sont souvent versionnées (Git) et servent de « source of truth » pour les pipelines de données. | Un script `extract_sales.sql` qui est appelé par un DAG Airflow. |

### 2️⃣ Est‑ce **indispensable** ?  

| **Situation** | **SQL indispensable ?** | **Pourquoi / alternatives** |
|---------------|--------------------------|-----------------------------|
| **Entreprise avec data‑warehouse relationnel** (ex. : Snowflake, Redshift, BigQuery) | **Oui** (ou au moins un langage de requête similaire). | La plupart des données sont stockées en tables ; sans SQL, il faut passer par des outils ETL lourds. |
| **Data Lake / fichiers plats (CSV, Parquet, JSON) stockés sur le cloud** | **Pas forcément**. | On peut lire directement avec `pandas.read_parquet`, `spark.read.json`, etc. Mais même dans ces cas, on utilise souvent **Spark‑SQL** ou **DuckDB** qui utilisent la même syntaxe que SQL. |
| **Projet de recherche académique** (datasets publics, Kaggle) | **Non**. | Les jeux de données sont souvent fournis sous forme de fichiers CSV/TSV que l’on charge directement en Python/R. |
| **Environnement de production MLOps** | **Oui**, mais parfois via des **ORM** ou des **API** qui encapsulent le SQL. | Bibliothèques comme SQLAlchemy (Python) ou dbt permettent de générer du SQL sans l’écrire à la main, mais la logique sous‑jacente reste du SQL. |
| **Petite startup sans base de données relationnelle** | **Non** (ou très peu). | Les données peuvent être dans NoSQL (MongoDB), dans des feuilles Google, ou dans des services SaaS qui exposent des API. |

> **Conclusion** :  
> - **Dans la majorité des postes Data Scientist en entreprise**, SQL est **pratiquement indispensable** pour accéder, filtrer et agrég­er les données.  
> - **Dans les contextes académiques ou de compétition**, on peut se passer de SQL, mais il reste très utile pour développer une bonne culture de manipulation de données.
